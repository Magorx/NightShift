## B. Architecture

### B1. GameManager is a god object (550+ lines, 40+ public fields)
- **Location**: `scripts/autoload/game_manager.gd`
- **Problem**: Holds building registry, placement/removal, deposit management, item visuals, terrain data, currency, delivery tracking, icon atlas, building costs, item definitions, placement configs, hotkeys, and references to every game system. Every system has a transitive dependency on every other through GameManager.
- **Fix**: Extract into: `BuildingRegistry` (defs, recipes), `GridState` (buildings dict, deposits, walls, terrain), `ItemRegistry` (item defs, icons, atlas), `PlacementSystem` (place/remove/can_place). GameManager becomes a thin facade.
- **Risk**: High | **Effort**: Large

### B2. Two competing ground-item systems (GroundItem vs PhysicsItem)
- **Location**: `player/ground_item.gd`, `scripts/game/physics_item.gd`
- **Problem**: `_spawn_ground_item()` was rewritten to spawn PhysicsItems, but GroundItem still exists, still in group `"ground_items"`, still referenced in save_manager for deserialization. Player pickup checks both groups. Two parallel systems doing the same thing.
- **Fix**: Migrate save deserialization to PhysicsItem. Delete GroundItem. Unify to one group.
- **Risk**: Medium | **Effort**: Medium

### B3. BuildSystem extends Node2D in a 3D scene
- **Location**: `scripts/game/build_system.gd:2`
- **Problem**: `class_name BuildSystem extends Node2D` but creates MeshInstance3D objects, uses 3D raycasts, adds children to Node3D layers. Vestigial from 2D era.
- **Fix**: Change to `extends Node3D` or `extends Node`.
- **Risk**: Medium | **Effort**: Small

### B4. Hotbar directly accesses inventory_panel's private members
- **Location**: `scripts/ui/hotbar.gd:100-101`
- **Problem**: Reads `_held_source_slot`, `_held_item`, `_selected_slots` (underscore-prefixed = private).
- **Fix**: Add public query methods: `is_slot_source_empty(index)`, `is_slot_selected(index)`.
- **Risk**: Medium | **Effort**: Small

### B5. UI code calls `player._spawn_ground_item()` (private method)
- **Location**: `scripts/ui/building_popup.gd:548`, `scripts/ui/inventory_panel.gd:363,375`
- **Problem**: UI panels calling private player methods is a layering violation.
- **Fix**: Make `spawn_ground_item()` public (drop underscore) since it has 3 external callsites.
- **Risk**: Low | **Effort**: Small

### B6. `buildings_panel._is_building_mode()` walks scene tree with string paths
- **Location**: `scripts/ui/buildings_panel.gd:296-302`
- **Problem**: `get_tree().root -> "GameWorld" -> "BuildSystem" -> building_mode`. Fragile string coupling.
- **Fix**: Add a query method on GameManager or BuildSystem.
- **Risk**: Medium | **Effort**: Small

### B7. `building_def.gd` instantiates scene twice during extraction
- **Location**: `buildings/shared/building_def.gd:59,119`
- **Problem**: `extract_from_scene()` instantiates once for shape (frees it), then again in `_extract_logic_script`. Doubles scene load time per building type.
- **Fix**: Combine into single instantiate + scan pass.
- **Risk**: Low | **Effort**: Small

## E. Performance

### E1. `update_cell()` rebuilds ENTIRE terrain mesh for a single tile change
- **Location**: `scripts/game/terrain_visual_manager.gd:110-118`
- **Problem**: Calls `build(map_size, ...)` which iterates all `map_size * map_size` cells. On 128x128 map = 16,384 cells. Called on every deposit depletion.
- **Fix**: Batch cell updates with a dirty flag, rebuild once per frame. Or partial mesh update for affected tile + neighbors.
- **Risk**: High | **Effort**: Medium

### E2. ItemIcon._process() runs on EVERY icon EVERY frame
- **Location**: `scripts/ui/item_icon.gd:57-78`
- **Problem**: Every ItemIcon polls mouse position, tooltip timing, RMB hold every frame. With 80+ icons visible (recipe menu + inventory + hotbar + popup), that's 80+ _process calls/frame.
- **Fix**: Use `mouse_entered`/`mouse_exited` signals. `set_process(false)` by default, enable on hover.
- **Risk**: Medium | **Effort**: Medium

### E3. `conveyor.gd` does string-path lookup every physics frame
- **Location**: `buildings/conveyor/conveyor.gd:40-61`
- **Problem**: `get_node_or_null("ForceZone")` runs at 60Hz per conveyor. With 200+ conveyors, that's 12,000+ string lookups/sec.
- **Fix**: Cache `force_zone` reference in `configure()` or `_ready()`.
- **Risk**: Low | **Effort**: Small

### E4. `PhysicsItem` allocates new PhysicsMaterial per item
- **Location**: `scripts/game/physics_item.gd:122-126`
- **Problem**: Every PhysicsItem creates a new identical PhysicsMaterial in `_ready()`.
- **Fix**: Make it a `static var _shared_material` initialized once.
- **Risk**: Low | **Effort**: Small

### E5. `ConcavePolygonShape3D` per mesh instance is expensive at scale
- **Location**: `buildings/shared/building_base.gd:54`
- **Problem**: `create_trimesh_shape()` on every MeshInstance3D on deferred ready. 200+ conveyors = heavy collision mesh generation.
- **Fix**: Use ConvexPolygonShape3D (acceptable approximation) or cache per building type in BuildingDef.
- **Risk**: Medium | **Effort**: Medium
