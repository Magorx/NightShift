## Conveyor & Transport System

### ~~1. Conveyor get_overlapping_bodies()~~ allocates a new Array every physics tick~~
- **Location**: `buildings/conveyor/conveyor.gd:48`
- **Problem**: `_force_zone.get_overlapping_bodies()` returns a freshly allocated Array every call. With 200+ conveyors at 60Hz, that's 12,000+ Array allocations/sec. Most conveyors are empty most of the time — they still allocate the array.
- **Proposed fix**: Track overlap count via `body_entered`/`body_exited` signals (increment/decrement a counter). Skip `get_overlapping_bodies()` entirely when the zone is empty. This avoids the allocation for the majority of conveyors that have no items.
- **Impact**: HIGH
- **Effort**: Small

### ~~2. Splitter re-computes output directions every physics tick~~
- **Location**: `buildings/splitter/splitter.gd:31, 51-66`
- **Problem**: `_get_output_directions()` iterates OutputZone children, reads positions, computes `global_transform.basis * dir` every `_physics_process` call. Splitters are static after placement — directions never change.
- **Proposed fix**: Cache the output directions array in `configure()`. Return the cached array from `_get_output_directions()`.
- **Impact**: MEDIUM
- **Effort**: Small

### ~~3. Splitter re-looks up ForceZone via string path every physics tick~~
- **Location**: `buildings/splitter/splitter.gd:27`
- **Problem**: `get_parent().get_node_or_null("ForceZone")` called every `_physics_process`. Already found in `_ready()` (line 19) but stored in a local variable instead of a member.
- **Proposed fix**: Store `_force_zone` as a member variable in `_ready()`, reference it in `_physics_process`.
- **Impact**: LOW-MEDIUM
- **Effort**: Small

### ~~4. Splitter get_overlapping_bodies() allocates a new Array every tick (same as #1)~~
- **Location**: `buildings/splitter/splitter.gd:35`
- **Problem**: Same as conveyor issue #1 — `force_zone.get_overlapping_bodies()` allocates every tick. Splitters already track items via `_assigned` dict, so the empty check is even easier.
- **Proposed fix**: Early-exit when `_assigned.is_empty()` AND no new bodies have entered since last check. Or use a counter like conveyors.
- **Impact**: MEDIUM
- **Effort**: Small

### ~~5. PhysicsItem uses Continuous Collision Detection (CCD) on every item~~
- **Location**: `scripts/game/physics_item.gd:32`
- **Problem**: `continuous_cd = true` on every PhysicsItem. CCD is expensive per-body — it runs additional swept-shape tests every physics step. Items are small, slow-moving objects on conveyors; they don't need CCD.
- **Proposed fix**: Remove `continuous_cd = true`. Items won't tunnel through anything at conveyor speeds (2.5 tiles/s). If specific fast-moving items need it later, enable selectively.
- **Impact**: HIGH
- **Effort**: Small

### ~~6. No PhysicsItem pooling — every spawn creates a new RigidBody3D~~
- **Location**: `scripts/game/physics_item.gd:134-147`
- **Problem**: Every `PhysicsItem.spawn()` creates a new RigidBody3D, generates a convex hull from mesh vertices, and adds it to the scene tree. The convex hull generation (`_gather_vertices` + `ConvexPolygonShape3D.new()`) is especially expensive. With items being constantly produced and consumed, this creates GC pressure and frame spikes.
- **Proposed fix**: Implement an object pool. On "despawn", hide and disable physics instead of `queue_free()`. On spawn, reuse a pooled item of the same `item_id`. Cache the collision shape per item type (already caches model scene but not the shape).
- **Impact**: HIGH
- **Effort**: Large

## Building Tick System

### ~~7. BuildingTickSystem.remove_at(i) causes O(n) array shift~~
- **Location**: `scripts/game/building_tick_system.gd:25`
- **Problem**: When a freed building is found mid-iteration, `buildings.remove_at(i)` shifts all subsequent elements. With 300 buildings and a removal in the middle, that's ~150 element copies.
- **Proposed fix**: Use swap-with-last-and-pop: `buildings[i] = buildings.back(); buildings.pop_back()`. Safe because iteration is already reverse.
- **Impact**: LOW
- **Effort**: Small

### ~~8. Converter scans InputZone children via string path every physics tick~~
- **Location**: `buildings/smelter/converter.gd:65-68`
- **Problem**: `_try_consume_inputs()` does `get_parent().get_node_or_null("Inputs")` then iterates `inputs_node.get_children()` with type checks every physics tick. These references never change after placement.
- **Proposed fix**: Cache the InputZone array in `configure()`. Iterate the cached array in `_try_consume_inputs()`.
- **Impact**: MEDIUM
- **Effort**: Small

### ~~9. InputZone.get_items() does validity scan with mid-array remove_at every call~~
- **Location**: `buildings/shared/input_zone.gd:18-26`
- **Problem**: Every `get_items()` call walks the entire `_overlapping_items` array checking `is_instance_valid()`. Invalid entries trigger `remove_at(i)` which is O(n) for array shift. Called by every converter and sink every physics tick.
- **Proposed fix**: The `_on_body_exited` callback (line 60-62) already removes items on exit. The `is_instance_valid` check is redundant safety — Godot fires `body_exited` when a body is freed. Remove the validity scan loop; just return `_overlapping_items` directly. Keep a debug assert if needed.
- **Impact**: MEDIUM
- **Effort**: Small

### ~~10. OutputZone._random_point_in_box iterates children to find CollisionShape3D every spawn~~
- **Location**: `buildings/shared/output_zone.gd:34-39`
- **Problem**: `_random_point_in_box()` iterates `get_children()` to find CollisionShape3D every item spawn. This is per-production-cycle, not per-frame, but still wasteful.
- **Proposed fix**: Cache the CollisionShape3D reference in `_ready()`.
- **Impact**: LOW
- **Effort**: Small

## Build System

### ~~11. BuildSystem does a physics raycast every frame (doubled during drag)~~
- **Location**: `scripts/game/build_system.gd:79`
- **Problem**: `_get_grid_pos_under_mouse()` calls `GridUtils.raycast_mouse_to_grid()` every `_process` frame. During drag, `_update_blueprints()` (line 81) triggers additional raycasts. The raycast hits a ConcavePolygonShape3D terrain with 16K+ triangles.
- **Proposed fix**: Cache the raycast result and skip when the mouse hasn't moved since last frame. Track `get_viewport().get_mouse_position()` and early-return on same position.
- **Impact**: MEDIUM
- **Effort**: Small

### ~~12. BuildSystem._update_destroy_highlights recomputes every frame in destroy mode~~
- **Location**: `scripts/game/build_system.gd:85`
- **Problem**: In destroy mode, `_update_destroy_highlights()` runs every `_process` frame even when the mouse hasn't moved. During drag it iterates every cell in the selection rectangle.
- **Proposed fix**: Only recompute when `cursor_grid_pos` has changed. Add a dirty flag.
- **Impact**: MEDIUM (during drag)
- **Effort**: Small

### ~~13. BuildSystem._update_ghosts re-validates placement for every blueprint every frame~~
- **Location**: `scripts/game/build_system.gd:81` (via _update_blueprints)
- **Problem**: During drag, every ghost calls `GameManager.can_place_building()` which iterates the building's rotated shape, checks walls dict, checks buildings dict. For 20 conveyors in a line, that's 20 validations per frame.
- **Proposed fix**: Cache validation results and only recompute when `cursor_grid_pos` changes.
- **Impact**: MEDIUM (during long drags)
- **Effort**: Medium

## Physics / Collision

### ~~14. Buildings use ConcavePolygonShape3D (trimesh) for collision~~
- **Location**: `buildings/shared/building_base.gd:50-63`
- **Problem**: Every building generates `ConcavePolygonShape3D` from its model meshes via `mesh.create_trimesh_shape()`. Trimesh shapes are the most expensive collision type in Godot physics — each item-vs-building collision check does triangle-level tests. With 200 buildings and 100 items, the broadphase pairs explode.
- **Proposed fix**: Replace trimesh with simple box/convex shapes per building type. Most buildings are roughly box-shaped. Define collision shapes manually in the `.tscn` or generate simplified convex hulls instead of trimesh. Trimesh should only be used for static-vs-static (terrain), not for dynamic item collisions.
- **Impact**: HIGH
- **Effort**: Medium

## UI System

### ~~15. ItemIcon._process runs on every instance every frame~~
- **Location**: `scripts/ui/item_icon.gd:57-78`
- **Problem**: Every ItemIcon instance (50+ in inventory, hotbar, popups, menus) runs `_process` every frame. It calls `_is_hovered()` which does `is_visible_in_tree()`, `get_viewport().get_mouse_position()`, `get_global_rect().has_point()`. Most icons have `mouse_filter = MOUSE_FILTER_IGNORE` (line 29) and can never be hovered.
- **Proposed fix**: Disable `_process` by default (`set_process(false)` in `_ready`). For browsable icons, enable via `mouse_entered`/`mouse_exited` signals. Non-browsable icons don't need per-frame processing at all.
- **Impact**: MEDIUM
- **Effort**: Small

### ~~16. ItemIcon._is_covered_by_window does full scene tree traversal per frame~~
- **Location**: `scripts/ui/item_icon.gd:104-140`
- **Problem**: `_collect_game_windows()` recursively walks the ENTIRE scene tree from root every frame (cache invalidated each frame via `process_frame` one-shot signal, line 114). With 200+ buildings (multi-node hierarchies), this traverses thousands of nodes.
- **Proposed fix**: Add GameWindow to a group (`"game_windows"`) in its `_ready()`. Use `get_tree().get_nodes_in_group("game_windows")` — O(1) Godot group lookup vs O(N) manual tree walk.
- **Impact**: MEDIUM-HIGH
- **Effort**: Small

### ~~17. HUD._update_phase_display rebuilds strings and overrides theme color every frame~~
- **Location**: `scripts/ui/hud.gd:131-146`
- **Problem**: Every frame: `"Round %d"` string format, `get_phase_name().to_upper()` string alloc, `"%02d:%02d" % [mins, sec_part]` string + array alloc, and `add_theme_color_override()` (dict insert + notification). Phase/round don't change every frame.
- **Proposed fix**: Only update round/phase labels in `_on_phase_changed()`. Only update timer label when the displayed seconds value changes (cache `_last_displayed_seconds`).
- **Impact**: LOW
- **Effort**: Small

### ~~18. HUD._on_node_added fires for every node added to the scene tree~~
- **Location**: `scripts/ui/hud.gd:34, 47-49`
- **Problem**: `get_tree().node_added.connect(_on_node_added)` fires for every node added to the entire scene tree. Building placement adds 5-15 nodes per building. Loading a save with 200 buildings fires hundreds of callbacks.
- **Proposed fix**: Set `focus_mode = FOCUS_NONE` on the project theme's BaseButton style. Remove the global `node_added` connection.
- **Impact**: LOW-MEDIUM (worst during load)
- **Effort**: Small

### ~~19. FPS label string formatting every frame~~
- **Location**: `scripts/ui/hud.gd:61`
- **Problem**: `"FPS: %d" % Engine.get_frames_per_second()` allocates a new string every frame.
- **Proposed fix**: Update FPS label every 0.5s using a frame counter or timer.
- **Impact**: LOW
- **Effort**: Small

## Terrain

### 20. Terrain update_cell rebuilds the ENTIRE mesh
- **Location**: `scripts/game/terrain_visual_manager.gd:111-126`
- **Problem**: `update_cell()` triggers `_deferred_rebuild()` which calls `build()` — iterating all 128x128 = 16,384 cells, generating vertices with SurfaceTool, and committing a new ArrayMesh. This happens when deposits deplete.
- **Proposed fix**: Currently acceptable (rare event). If Night Shift adds terrain destruction during fights, implement chunked terrain (e.g., 16x16 chunks) so only the affected chunk rebuilds.
- **Impact**: LOW now (rare event), HIGH if terrain changes become frequent
- **Effort**: Large
